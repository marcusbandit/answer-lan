What Is a Semaphore in Concurrency?

A semaphore is a synchronization primitive that controls access to a shared resource using a counter. Threads call wait() (also called P or acquire) to decrement the counter and enter, and signal() (also called V or release) to increment it and leave. If the counter is zero, any thread calling wait() blocks until another thread calls signal().

How it works

A semaphore wraps an integer count plus a queue of waiting threads. The two operations are atomic:

  • wait() / acquire(): if count > 0, decrement it and proceed. If count == 0, block until it's signaled.
  • signal() / release(): increment count, and wake one waiting thread if any are queued.

Counting vs. binary semaphores

TypeCounter rangeTypical use
Counting semaphore 0 to N Limiting concurrent access to a pool of N identical resources (e.g. N database connections, N worker slots)
Binary semaphore 0 or 1 Acts like a lock, only one holder at a time

Semaphore vs. mutex

They look similar but solve different problems:

MutexSemaphore
Purpose Mutual exclusion (one owner) Signaling / limiting concurrency (N slots)
Ownership Only the thread that locked it may unlock it Any thread can call signal(), even one that never called wait()
Typical count Always 1 Any non-negative integer
Common use Protecting a critical section Producer-consumer queues, connection pools, rate limiting

Example: limiting concurrent workers

semaphore sem = Semaphore(3)  // only 3 workers at a time

function worker(job):
    sem.wait()        // acquire a permit, blocks if 3 already active
    try:
        process(job)
    finally:
        sem.signal()  // release the permit

Even if 100 jobs start at once, only 3 will run process(job) concurrently; the rest queue on wait() until a permit is freed.

Where semaphores show up

  • Producer-consumer problems: one semaphore tracks filled slots in a buffer, another tracks empty slots.
  • Connection/thread pools: cap how many callers can use a limited resource simultaneously.
  • Rate limiting: permits represent "requests allowed right now."
  • Signaling between threads: unlike a mutex, a semaphore can be signaled by a thread that never waited on it, which makes it useful for "notify another thread an event happened" patterns.

Invented by Edsger Dijkstra in 1965 for the THEOS operating system; the names P (from Dutch proberen, "to try") and V (verhogen, "to increase") for wait/signal come from that original work.
